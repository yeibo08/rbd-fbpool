# Football Pool — Spec v2

> This document supersedes SPEC.md. It incorporates all clarifications, corrections, and enhancements introduced during implementation and the improvement plan.

---

## What is a football pool

A football pool is a competitive framework where users predict match outcomes to earn points. This spec outlines a system for managing entries, scoring results, and tracking real-time leaderboards for the FIFA World Cup 2026 (June 11 – July 19, 2026, 104 matches, 48 teams).

---

## Core Concepts

**User** A registered participant with a unique idempotent identifier (nanoid). Has a display name, email (stored lowercase), and an encrypted password. Belongs to one or more groups of friends. May have a `forcePasswordChange` flag set after a system-generated temporary password is issued.

**Group of friends** A group of up to 20 users sharing a leaderboard. Has a unique idempotent identifier and an invite token used to generate shareable join links. Each member holds one of three roles: owner, manager, or member. Deleting a group cascades to all related membership rows, scoring rules, and predictions.

**Roles**
- *Owner* — creates the group; can change member roles, remove members, delete the group, and manage scoring rules.
- *Manager* — can manage scoring rules, remove members, and regenerate the invite link.
- *Member* — participates in predictions; cannot manage the group.

**Pointing rules** Per-group scoring configuration that determines how many points are awarded for each correct prediction criterion per match. The four criteria are: correct match result (win/draw/loss), exact home team goals, exact away team goals, and exact total goals. All four values default to 1. Rules can be changed by the owner or any manager at any time.

**Fixture** The complete match calendar for FIFA World Cup 2026 — 104 matches, 48 teams, 12 groups. Seeded from `fixture.md`. Includes group stage (72 matches), round of 32 (16), round of 16 (8), quarterfinals (4), semifinals (2), third-place play-off (1), and final (1).

**Match** A scheduled game between two teams. Has a kickoff time, venue, stage, and optional group letter. Includes a `deadlineAt` field equal to `kickoffAt − 15 minutes`, after which predictions are locked. After the match, the official result (home goals, away goals, penalty info) is stored and predictions are scored.

**Prediction** A user's score guess (home goals, away goals) for a specific match within a specific group. Can be created or updated any number of times until the deadline. Each prediction belongs to exactly one user + group + match combination.

**Leaderboard** A ranked list of group members sorted by total accumulated points across all completed matches, computed on request. Members with equal points share the same rank. The user with the most points at the end of the tournament is the winner.

**Invite link** A shareable URL of the form `/unirse/:token` that lets any registered user join a group. The token can be regenerated by the owner or a manager, invalidating the previous link.

**Prediction progress** A per-group count of how many matches a user has predicted out of the total. Shown on the groups list as "X / 104 pronosticados".

---

## Winner

The user with the most accumulated points at the end of the tournament, as computed by the group's scoring rules across all 104 matches, is the winner of that group.

## Features

### Authentication

Users must register with a valid email address and a password that meets all of the following:
- At least 8 characters
- At least 1 uppercase letter
- At least 1 digit
- At least 1 of the symbols: `! # $ ?`

Passwords are stored hashed (bcrypt, 10 rounds). Authentication uses a JWT stored in an httpOnly cookie (15-minute expiry). The token is verified on every protected request via middleware.

**Forgot password** — submitting a registered email causes the system to generate a new temporary password that satisfies all password rules. The temporary password is returned in the response (only once). On the next login with this password, `forcePasswordChange` is set to `true` and the client is redirected to the change-password screen before any further navigation is allowed. After a successful password change, `forcePasswordChange` is cleared.

**Login redirect** — if a user reaches the login screen from a protected page (e.g. an invite link), after logging in they are redirected back to that page rather than to the default groups list.

---

### User profile

Each user can:

- Edit their display name — via the hamburger menu drawer (`PATCH /api/auth/me`); re-issues a fresh JWT so the cookie reflects the new name immediately
- Change their password (requires the current password)
- View all groups they belong to
- Generate a QR code encoding their profile URL, which a group admin can scan to add them directly to a group

---

### Groups of friends

A user can create a group (becoming its owner) or join one via invite link or QR scan.

**Invite link** — each group has a unique token-based invite URL. The owner or any manager can regenerate the token, invalidating the old link. A logged-out user who opens an invite link is redirected to login and returned to the invite link afterward to complete the join.

**Member limit** — a group accepts at most 20 members. Attempts to join a full group are rejected with a clear error.

**Rename group** (owner or manager) — the group name can be changed at any time via `PATCH /api/groups/:id`. The new name is validated (1–60 characters). The change is reflected immediately in all views.

**Member management** (owner only) — promote a member to manager or demote a manager back to member. Remove any non-owner member.

**Group deletion** (owner only) — permanently deletes the group and cascades to all related records: member rows, scoring rules, and all predictions made within the group.

---

### Customized scoring rules

The owner or any manager can configure four point values per group:

| Criterion | Description |
|---|---|
| Correct result | Predicted the right outcome (home win, away win, or draw) |
| Exact home goals | Predicted the exact number of goals for the home team |
| Exact away goals | Predicted the exact number of goals for the away team |
| Exact total goals | Predicted the exact sum of goals for both teams |

Default value for each criterion is 1 pt. Values may be 0 (to disable a criterion) or any positive integer. The maximum points per match equals the sum of all four values.

**Penalty shootouts** — for knockout matches that go to penalties, the team that wins the shootout is treated as the match winner for the purpose of the "correct result" criterion. Goals are always regulation + extra time only; penalty kick goals are excluded from all goal-based criteria.

Rules can be updated at any time and take effect on all future leaderboard recalculations.

---

### Match center

Each group has its own match center showing all 104 matches grouped by calendar day, sorted by kickoff time. Each match card displays:
- Home and away team names
- Stage and group letter (if applicable)
- Local kickoff time (user's local timezone)
- The user's current prediction (if any)
- The official result and points earned (once available)

**Filters** — users can filter the match list by: All / Open (deadline not yet passed) / Finished (result available). Each filter tab shows a friendly message when no matches match the criteria.

**Prediction input** — for open matches, the user enters home and away goals and saves. Changes are allowed at any time before the deadline. Saves update the UI immediately (optimistic update) without a loading spinner.

**Deadline countdown** — when a match closes within 60 minutes, a countdown pill ("Cierra en X:XX") is shown below the prediction input and ticks every second.

**Locked state** — between the deadline and the final result, the user's saved prediction is shown as read-only with a "(cerrado)" label.

**Deadline enforcement** — the server rejects prediction submissions received at or after `kickoffAt − 15 minutes`, regardless of client-side state.

---

### Results sync

Official match results are fetched automatically from the ESPN public API (no API key required) via a background job that runs every 5 minutes on match days. A match is eligible for sync when its kickoff was at least 90 minutes ago and it has no recorded result.

An alternative provider (football-data.org, free tier) can be configured via environment variable as a fallback. The provider abstraction allows swapping the data source without changing any other code.

Sync failures for individual matches are logged and skipped without interrupting the job — remaining matches in the batch are still processed.

---

### Leaderboard

The leaderboard is computed on request for each group. It shows all members ranked by total points, with ties sharing the same rank. Each row shows display name, role, total points, and rank.

Points are computed using `computePoints(prediction, result, rules)`:
- If the match went to penalties, the penalty winner determines the "correct result" outcome
- Penalty kick goals are excluded from home, away, and total goal checks
- A user who did not predict a match earns 0 points for it

---

### Prediction progress

The groups list page shows a per-group progress counter below each group name: "X / 104 pronosticados". This reflects how many of the 104 matches the current user has submitted a prediction for in that group.

---

### Navigation

All authenticated pages share a common `AppNav` component that replaces per-page navbars. It provides:

- **Left side** — breadcrumb trail indicating the current position in the page hierarchy:
  - Grupos: app title only
  - GroupDetail: ← Grupos / [group name]
  - MatchCenter: ← Grupo / Partidos
  - Leaderboard: ← Grupo / Tabla de posiciones
- **Right side** — a hamburger icon (☰) that opens a slide-out drawer containing:
  - The user's display name with an inline edit field (triggers `PATCH /api/auth/me`)
  - Links to Mis grupos and Ayuda/FAQ
  - Logout button

The drawer overlays the page with a dark backdrop and is dismissed by tapping outside it or the close button.

### FAQ

A public page at `/faq` (no login required) with accordion-style Q&A covering:

2. How points are calculated
3. When predictions close
4. Whether predictions can be changed
5. Penalty shootout rules

The page is linked from the hamburger drawer (authenticated users) and as a footer link on the Login and Register pages.

## Non-functional requirements

- All UI text in Spanish
- Responsive layout targeting mobile (375 px) and desktop (1280 px)
- All timestamps stored and transmitted as UTC ISO 8601 strings
- Passwords never stored or logged in plain text
- The admin routes (`PUT/DELETE /api/admin/matches/:id/result`) that allow manually setting results are only mounted outside of `NODE_ENV=production`
- Cascade deletes enforced at the database schema level (`ON DELETE CASCADE`)
- Shared authentication utility (`requireMember`) prevents access to any group resource by non-members at the route handler level
- Deployed to Fly.io via Docker; SQLite persisted on a named volume at `/data/pool.db`; app scales to zero when idle (`min_machines_running = 0`) and wakes on first request
- CI/CD via GitHub Actions: every push to `main` triggers `flyctl deploy --remote-only`
- In production the Hono server serves the compiled Vite client as static files and handles SPA routing with an `index.html` fallback; no separate static host needed
